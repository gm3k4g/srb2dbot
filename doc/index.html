srb2dbot

A Discord bot that remotely controls Sonic Robo Blast 2 game servers. Built with the D++ library in C++26, it manages server lifecycle, script editing, WAD file uploads, in-game administration, and real-time chat bridging - all through Discord.

Quick Start

# Clone and build
git clone https://github.com/gm3k4g/srb2dbot.git
cd srb2dbot

# On NixOS / with Nix installed
./build.sh

# On regular Linux
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j$(nproc)

# Configure
cp secret.json.default secret.json
# Fill in the fields described below

Deployment (NixOS → Standard Linux)

On NixOS, the binary's ELF interpreter points to /nix/store/.../ld-linux-x86-64.so.2. Standard Linux puts it at /lib64/. Fix with: 

# Build release (statically links C++ runtime)
./build.sh --release

# Fix ELF interpreter for portability
nix-shell -p patchelf --run "patchelf --set-interpreter /lib64/ld-linux-x86-64.so.2 build/srb2dbot"

# SCP to server (target needs libdpp-dev, libssl3, libopus0)
scp build/srb2dbot user@server:~/.srb2dbot/

Configuration

Create secret.json from the template:

{
    "bot_token":     "YOUR_BOT_TOKEN",
    "bot_id":        "YOUR_BOT_USER_ID",
    "guild_id":      "YOUR_GUILD_ID",
    "channel_id":    "YOUR_BRIDGE_CHANNEL_ID",
    "service_name":  "srb2@srb2b"
}
FieldRequiredDescription
bot_token Yes Discord bot token from Developer Portal → Bot → Token
bot_id Yes The bot's own Discord user ID (Application ID). Prevents the chat bridge from echoing its own messages
guild_id Yes Discord server (guild) ID where slash commands are registered
channel_id For bridge Discord channel ID for chat relay. If unset or "0", the bridge is disabled
service_name For server control systemd user service name. Defaults to srb2@srb2b

Important: Enable the Message Content Intent in the Discord Developer Portal under Bot → Privileged Gateway Intents for the chat bridge to read message content.

How It Works

The bot communicates with an SRB2 server through a named FIFO pipe (~/.srb2/srb2_servers.d/srb2b.d/srb2b.fifo). SRB2 reads commands from this pipe as if they were typed in the server console.

Server lifecycle (start, stop, restart) is managed through systemd user services, with the service name configurable via secret.json.

Script editing operates on the bash configuration file (~/.srb2/srb2_servers.d/srb2b.d/srb2b.sh), validated with bash -n before writing.

The chat bridge relays messages between Discord and SRB2 in real time via shared text files in ~/.srb2/luafiles/client/DiscordBot/. Discord messages are sanitized (ASCII-filtered, links replaced) and forwarded to SRB2. SRB2 chat is polled every 2 seconds and posted to Discord with emoji conversion and mention suppression. Gametype names in events use SRB2's internal G_GetGametypeName API, ensuring exact 1-to-1 names ("Co-op", "Arena") and automatic support for custom gametypes from WAD files (e.g. "Survival"). Events are flushed to Messages.txt immediately when they occur rather than waiting for the periodic poll cycle. On bot startup, a dbot_sync command requests current server state re-emission, so events appear within ~2 seconds even after a restart.

Security Model

  • All slash commands require p_ban_members permission
  • Pipe input is sanitized - newlines, carriage returns, and control characters are rejected
  • Script validation uses fork/exec with argument arrays instead of shell string construction
  • Uploaded files are capped at 100 MB with filename path-traversal protection
  • Password lines in scripts are redacted automatically

Dependencies

srb2dbot (build)

  • C++26 compiler (GCC 14+ or Clang 18+)
  • CMake 3.14+
  • D++ 10.x - Discord API library
  • nlohmann/json 3.x - JSON parsing
  • systemd - for service management
  • bash - for script validation

Companion tools (runtime)

  • ImageMagick (imagemagick) - Thumbnail PPM→PNG conversion, intermission overlay generation
  • unzip - PK3 lump extraction (thumbnail grabber only)

Test Suite

The project includes 77 tests (66 unit + 11 end-to-end pipeline tests) covering utility functions, filename sanitization, script manipulation, chat bridge logic (sanitization, file polling, emoji conversion, event parsing), polling lifecycle, and full event pipeline simulation. Run with: 

./build.sh --test
