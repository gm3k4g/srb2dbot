Runtime Pipeline

Commands and messages flow through four distinct paths:

1. Script Path

Script editing commands (/get_script, /find_line, /insert_line, etc.) read and write ~/.srb2/srb2_servers.d/srb2b.d/srb2b.sh. Each write is validated with bash -n via fork() + execlp() before committing.

Core logic is extracted into source/script.cpp as pure string-in/string-out functions, making them independently testable without filesystem access.

2. WAD Path

WAD commands (/list_wads, /search_wads, /addfile_upload, /addfile_link) operate on ~/.srb2/addons/. Uploads are capped at 100 MB and filenames are sanitized to prevent path traversal attacks.

3. Server Pipe Path

Server and player commands (/server_do, /server_say, /kick_player, /ban_player, /restart_server, /stop_server) communicate with SRB2 through:

  • FIFO pipe at ~/.srb2/srb2_servers.d/srb2b.d/srb2b.fifo for console commands. All input is sanitized to reject control characters and newlines.
  • systemd via systemctl restart|stop <service> --user for service lifecycle. The service name is configurable in secret.json.

4. Chat Bridge Path

Real-time message relay between Discord and SRB2 via shared text files in ~/.srb2/luafiles/client/DiscordBot/:

  • Discord → SRB2: Messages from the configured bridge channel are sanitized (ASCII filter, newline escaping, link replacement) and written to discordmessage.txt for a companion Lua script to forward to SRB2.
  • SRB2 → Discord: A D++ timer polls Messages.txt every 2 seconds for new lines, converts :emoji: references to Discord format using cached guild emojis, suppresses all mentions, and posts to the bridge channel.

Gametype names in events are resolved through SRB2's G_GetGametypeName Lua API (scripts/SRB2DiscordBot-v0.1.35.lua:451), ensuring 1-to-1 accuracy with in-game names and automatic support for custom WAD gametypes (e.g. "Survival" from battle mod).

Events are flushed to Messages.txt immediately via flush_msgsrb2() rather than waiting for the periodic 70-tic cycle. Three flush paths exist:

  • Immediate (primary) - called after every event emission (SERVER_START, ROUND_START/END, PLAYER_JOIN/QUIT)
  • Periodic (fallback) - inside bot_function() every ~2 seconds
  • Message-triggered - via server_log msg when cv_messagedelay is disabled

On bot startup, a dbot_sync FIFO command triggers the Lua script to re-emit current server state (SERVER_START + ROUND_START if active). This ensures events are visible within the next poll cycle even after a restart.

Both files are cleared on bot startup. The bridge channel and bot identity are configured in secret.json.

Double-Post Prevention

The PlayerMsg hook in SRB2 can fire multiple times for a single chat message (engine-level behavior). The fires do not always occur at the same leveltime — they can span 1-3 different game ticks. The companion Lua script prevents duplicate Discord posts through three mechanisms:

  • File-local state tables (NetVar deep-copy immunity): _player_msg_cache, _pending_joins, _join_emitted, and _join_times are declared as local variables at the top of the script, captured by hook closures. File-local variables are NOT in _G and are immune to SRB2's NetVar deep-copy, which replaces the DiscordBot global table when a player joins/leaves. Without this fix, the deep-copy wiped the dedup cache and join guard, causing the engine multi-fire to pass through and produce duplicate CHAT events with different jointime values (jointime was re-set by the re-processed join ThinkFrame, 2 seconds apart). The duplicate PLAYER_JOIN was silently dropped by C++ seen_joins dedup (now removed), masking the root cause.
  • 5-tic dedup window in PlayerMsg: Each message is tracked by (player_node, type, target, msg) combined with leveltime. On a repeat fire within 5 tics (leveltime - cache[key] <= 5), true is returned to suppress the duplicate without writing to Messages.txt. At 35 tics/second, 5 tics ≈ 143ms — the engine multi-fire spans 1-3 tics (well within), while human typing is ≈250ms apart (≈8.75 tics, safely outside). A previous 10-tic window was too broad (suppressed repeats 250ms apart), and a same-tick check was too narrow (engine fires span different ticks).
  • Guard fix in server_log msg: The handler now checks DiscordBot.Data.msgsrb2 ~= '' alongside the existing truthy check, preventing file open/write/close cycles when the buffer is empty (empty string is truthy in Lua).

The C++ bot applies two additional measures:

  • No phantom newline: The bot no longer writes \n to Messages.txt on startup, eliminating the phantom empty line that was previously tracked as line 1.
  • No C++ dedup layers: Both the content-based CHAT/SERVER_CHAT dedup (seen_lines, 1-second window) and the PLAYER_JOIN dedup (seen_joins, 5-second window) were removed. These were masking the real duplication root cause by silently dropping duplicate events at the C++ level. With the Lua file-local variables immune to NetVar deep-copy, all dedup is now handled at the source in Lua.

Key Files

FileRole
source/main.cppBot entry point. Initializes D++, reads secrets, registers slash commands, runs the event loop. Contains anonymous namespace functions for filesystem operations that depend on POSIX APIs (FIFO pipes, directory iteration, process management).
source/utils.cppPure utility functions: sanitize_filename() strips path components from user input, link_filename_str() extracts filenames from URLs.
source/script.cppScript manipulation logic operating on string content rather than files. Includes script_inspect_str(), script_find_str(), script_move_line_str(), script_add_line_str(), script_remove_line_str(), and script_change_line_str().
source/bridge.cppChat bridge logic: message sanitization for Discord→SRB2 (sanitize_message_for_srb2(), ASCII filter, link replacement), file polling helpers (bridge_get_lines(), bridge_read_range()), and emoji conversion (bridge_replace_emojis()).
secret.json.defaultConfiguration template - copy to secret.json and fill in bot token, bot ID, guild ID, channel ID, and service name.
scripts/SRB2DiscordBot-v0.1.35.luaSRB2-side companion script that runs in-game. Handles chat forwarding, event emission (ROUND_START, ROUND_END, CTF_CAPTURE, etc.), immediate flush via flush_msgsrb2(), startup sync via dbot_sync command, and gametype name resolution via G_GetGametypeName() with fallback table.
source/version.h.cmakeCMake template that generates version.h with project metadata and command name strings.

Security Architecture

Attack VectorMitigation
Command injection via script pathfork() + execlp() with argument arrays - shell never interprets the path
Command injection via pipe inputControl characters (\x00-\x1F except tab), newlines, and carriage returns are rejected before writing to the FIFO
Path traversal via upload filenamessanitize_filename() extracts only the filename component via std::filesystem::path::filename(), rejects . and ..
Unbounded upload sizeDownloads are capped at 100 MB; response bodies exceeding MAX_WAD_SIZE are rejected
JSON parse exceptionstry/catch around json::parse() with early file-open check
Information disclosure (passwords)Lines containing "password" are redacted in script output
FIFO truncationwrite() retry loop handles partial writes and EAGAIN

CI Pipeline

CMake presets provide automated builds for Ubuntu (GCC/clang with clang-tidy, cppcheck), macOS (AppleClang), and Windows (MSVC). Dev mode enables CTest, clang-format linting, codespell, and code coverage with gcov/lcov.
